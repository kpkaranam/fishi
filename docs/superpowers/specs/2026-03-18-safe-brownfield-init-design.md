# Safe Brownfield Initialization

**Date:** 2026-03-18
**Status:** Approved
**Scope:** `packages/core` + `packages/cli`

## Problem

FISHI's `fishi init` blindly overwrites existing project files — `.claude/CLAUDE.md`, `.claude/settings.json`, `.mcp.json`, agents, skills, commands — without checking for conflicts. On brownfield projects with established Claude Code setups, this destroys user configuration.

## Core Principle

Existing files are sacred. FISHI never modifies or removes user files without explicit permission. Always backup before touching anything.

## Design

### Init Flow (Brownfield)

```
fishi init (brownfield detected)
  │
  ├─ 1. Brownfield analysis (unchanged — reads only)
  │
  ├─ 2. Re-init guard
  │     If .fishi/ exists: prompt "FISHI already initialized. Re-initialize? [y/N]"
  │     If yes: proceed to conflict detection (treat .fishi/ files as conflicts too)
  │     If no: exit with "Run fishi status to see project state"
  │
  ├─ 3. Conflict detection
  │     Scan all files FISHI wants to create against what exists
  │     Build a conflict map: { path, exists, category }
  │
  ├─ 4. Backup
  │     Snapshot ALL conflicting files to .fishi/backup/<ISO-timestamp>/
  │     Write manifest.json after backup completes (atomic — never partial)
  │
  ├─ 5. Interactive conflict resolution (grouped by category)
  │     For each category with conflicts, prompt:
  │       [S]kip  |  [M]erge  |  [R]eplace
  │     Default recommendation: Merge
  │
  └─ 6. Execute
        Apply user's choices per category
        Non-conflicting files created silently
```

### Non-Interactive / CI Mode

When `--no-interactive` or zero-config mode is used:
- **Default**: Skip all conflicting files, create only non-conflicting files
- **`--merge-all`**: Merge all conflicting files (backup still created)
- **`--replace-all`**: Replace all conflicting files (backup still created)

### Conflict Categories

Instead of per-file prompts for ~82 files, conflicts are grouped into 8 categories:

| # | Category | Prompt Example |
|---|----------|---------------|
| 1 | `.claude/CLAUDE.md` | "Found existing .claude/CLAUDE.md (14.8 KB). Merge adds a `### FISHI Framework` section." |
| 2 | `.claude/settings.json` | "Found existing .claude/settings.json with 3 hooks, 12 permissions. Merge adds FISHI hooks/permissions alongside yours." |
| 3 | `.mcp.json` | "Found existing .mcp.json with 7 servers. Merge adds FISHI entries without removing yours." |
| 4 | Agents | "Found 8 existing agents. FISHI adds 18 agents (no filename conflicts)." |
| 5 | Skills | "Found 5 existing skills. FISHI adds 12 skills (no filename conflicts)." |
| 6 | Commands | "Found 7 existing commands. FISHI adds 8 commands (no filename conflicts)." |
| 7 | Hooks | "Found 3 existing hook scripts. FISHI adds its hook scripts alongside yours." |
| 8 | `.gitignore` | "Found existing .gitignore. Merge appends FISHI entries (.trees/, .fishi/logs/)." |

Categories with no conflicts are skipped (no prompt).

### Merge Strategies

#### `.claude/CLAUDE.md`
- Read existing content
- Check if `### FISHI Framework` section already exists (idempotency)
  - If exists: replace only the FISHI section (between sentinel markers)
  - If not: append the section at the end
- Section format:
  ```markdown

  ---
  ### FISHI Framework
  <!-- FISHI:START — Auto-generated by FISHI. Do not edit between markers. -->
  [FISHI orchestration content here]
  <!-- FISHI:END -->
  ```
- Existing content above the markers is never modified

#### `.claude/settings.json`
- Union merge:
  - `hooks`: keyed by `(eventName, matcher, command)` triple. Add FISHI entries that don't match any existing triple. Never remove existing entries.
  - `permissions.allow`: union array (deduplicate exact strings)
  - `permissions.deny`: union array (deduplicate exact strings)
  - `env`: preserve existing keys, add new keys only
  - `enabledPlugins`: union array
  - Any other existing keys: preserve as-is

#### `.mcp.json`
- Parse existing MCP config
- For each FISHI MCP entry: add only if the server name key doesn't already exist
- Never remove existing entries

#### `.gitignore`
- Check if FISHI entries (`.trees/`, `.fishi/logs/`) already present
- If not: append FISHI block with comment header
- Never remove existing entries

#### Agents / Skills / Commands / Hooks
- Add FISHI files alongside existing ones
- If a filename collision exists (e.g., both have `code-reviewer.md`):
  - Prompt per-file within the category: **Skip** or **Replace** only
  - No "merge" option for agent/skill/command files (YAML frontmatter + markdown structure cannot be safely merged by appending)

### Backup Structure

```
.fishi/backup/
  2026-03-18T14-30-00/
    .claude/
      CLAUDE.md
      settings.json
      hooks/
        auto-format.sh
        block-dangerous-commands.sh
        session-log.sh
      agents/
        [all existing agent files]
      commands/
        [all existing command files]
      skills/
        [all existing skill dirs]
    .mcp.json
    .gitignore
    manifest.json
```

Backup is unconditional for ALL conflicting files regardless of the user's eventual choice (skip/merge/replace). The manifest records what was backed up but NOT the intended action (action is unknown at backup time).

`manifest.json` schema:
```json
{
  "timestamp": "2026-03-18T14:30:00.000Z",
  "fishi_version": "0.1.0",
  "files": [
    { "path": ".claude/CLAUDE.md", "size": 15123 },
    { "path": ".claude/settings.json", "size": 2048 }
  ]
}
```

Manifest is written atomically: write to `manifest.json.tmp`, then rename to `manifest.json`. A missing or absent manifest indicates an interrupted backup — init should abort and warn the user.

### Non-Conflicting Files (Silent Creation)

Everything under `.fishi/` is a new namespace — no prompts needed:
- `.fishi/state/`, `.fishi/scripts/`, `.fishi/memory/`
- `.fishi/todos/`, `.fishi/agent-factory/`, `.fishi/taskboard/`
- `.fishi/learnings/`, `.fishi/logs/`
- `.trees/`

`docs/README.md`: only created if `docs/README.md` does not already exist. If it exists, skip silently (not a conflict prompt — just don't overwrite).

### Greenfield Behavior

When no conflicts are detected (fresh project), the current behavior is preserved — all files created without prompts.

## New Modules

1. **`packages/core/src/generators/conflict-detector.ts`**
   - `detectConflicts(targetDir, fileList) → ConflictMap`
   - Scans target directory for each file FISHI wants to create
   - Groups results by category
   - Returns: `{ category, conflicts: [{ path, exists, size }], noConflicts: [path] }`

2. **`packages/core/src/generators/backup-manager.ts`**
   - `createBackup(targetDir, conflictingFiles) → backupPath`
   - Creates timestamped backup directory
   - Copies all conflicting files preserving directory structure
   - Writes manifest.json atomically (tmp + rename)
   - On failure: cleans up partial backup directory, throws with clear message

3. **`packages/core/src/generators/merge-strategies.ts`**
   - `mergeClaudeMd(existing, fishiContent) → merged` (idempotent — detects existing FISHI section)
   - `mergeSettingsJson(existing, fishiSettings) → merged` (union by triple key)
   - `mergeMcpJson(existing, fishiMcp) → merged` (add missing server names)
   - `mergeGitignore(existing, fishiAdditions) → merged` (append if missing)
   - Each strategy is a pure function: (existing, new) → merged

4. **Modified: `packages/core/src/generators/scaffold.ts`**
   - `write()` function accepts a resolution map
   - Skips files marked "skip"
   - Calls merge strategy for files marked "merge"
   - Overwrites for files marked "replace"
   - Writes directly for non-conflicting files
   - `docs/README.md`: skip if exists

5. **Modified: `packages/cli/src/commands/init.ts`**
   - Replace hard `.fishi/` exit guard with interactive re-init prompt
   - After brownfield analysis, run conflict detection
   - If conflicts found: backup → interactive prompts → pass resolution map to scaffold
   - If no conflicts: proceed as before (greenfield path)
   - Non-interactive mode: respect `--merge-all` / `--replace-all` flags, default to skip

## Testing

- Unit tests for conflict-detector (mock fs)
- Unit tests for each merge strategy (pure functions):
  - CLAUDE.md: first merge, idempotent re-merge, existing content preserved
  - settings.json: hooks union by triple, permissions dedup, existing keys preserved
  - .mcp.json: add new servers, skip existing names
  - .gitignore: append once, idempotent on re-run
- Unit tests for backup-manager:
  - Backup creates correct structure with manifest
  - Manifest is atomic (tmp + rename)
  - Interrupted backup: partial directory cleaned up
  - Backup can restore original content byte-for-byte
- Integration test: brownfield init with existing files (all merge strategies exercised)
- Integration test: greenfield init (regression — unchanged behavior)
- Integration test: re-init on already-initialized project
- Test: filename collision within a category (skip or replace only)
- Test: non-interactive mode with --merge-all and --replace-all flags
- Test: docs/README.md skipped when it already exists
